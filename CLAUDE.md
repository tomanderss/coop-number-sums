# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Number Sums logic puzzle PWA. Vanilla JS + Vue 3 via ESM — **no bundler, no transpiler, no TypeScript**. Deployed via GitHub Pages; also native iOS/Android via Capacitor (unchanged web app wrapped with `scripts/cap-copy-www.js` → `www/`).

## Commands

```bash
npm test                                        # unit + e2e
npm run test:unit                               # node --test only
npm run test:e2e                                # Playwright only
node --test test/unit/solver.test.js            # single unit file
npx playwright test test/e2e/gameplay.spec.js   # single e2e file
python3 -m http.server 8099                     # dev server (no npm script)
npm run cap:sync                                # copy www/ + cap sync (before native build)
node build.js                                   # cut release (bumps version, clears changes.txt)
node build.js --major                           # bump major, reset minor
```

## Architecture

| File | Role |
|---|---|
| `js/config.js` | Static config only: difficulties, colors, `DEFAULT_SETTINGS`. No logic. Difficulties also carry UI-only `accent`/`accentD`/`heat` (Leitfarbe + „Hitze" 0..1) für die **Slider-Schwierigkeitsauswahl**: EIN Slider schwach→stark, dessen Hintergrund über registrierte `@property`-Variablen (`--dacc`/`--dacc-d`/`--dheat` in styles.css, per `diffVars(id)` am Container gesetzt) SMOOTH morpht (registrierte Custom Props MÜSSEN in `transition-property` stehen — `background` allein animiert nicht; Bedienelemente laufen über die kurze `--snapT`, nicht die lange `--morphT`, sonst zieht der Regler nach). Medaillon-Icon + Name + Maße + Belohnung (Count-up)/Zeiten. Der Slider steckt in der **wiederverwendbaren `<difficulty-slider>`-Vue-Komponente** (app.js, `v-model`=Schwierigkeits-ID, Prop `coop` für Coop-Münzen×2/Lobby-Zeiten; enthält den **„Zufall"-Knopf** `randomize()`, der eine zufällige — immer ANDERE — Schwierigkeit wählt; wirkt via `v-model` in ALLEN Auswahl-Screens Solo+Coop/Race/Team). **ALLE Modi sehen identisch aus** (`.setup-slider`): Solo-Setup UND Coop/Race/Team-Host-Auswahl (Coop-Screen bekommt via `isCoopDiffView` die `.setup-slider`-Klasse; der Raumcode sitzt als `.setup-coderow` INNERHALB des Screens). Der morphende Hintergrund ist **full-bleed** (`.setup-slider::before` = `position:fixed`, auf die zentrierte App-Breite `--apphalf` geklemmt, wie `.winfx` nicht vom App-Padding beschnitten) und **blendet zu ALLEN vier Rändern** über zwei sich schneidende Achsen-Masken (`mask-composite:intersect`) weich aus — kein dunkler „Bilderrahmen". `.diff-card` (kompakte Karte) lebt nur noch für den sekundären Race-Rematch-Picker. Das alte `.opt-card`-Kartenraster ist raus. |
| `js/solver.js` | Deduction engine: `logicalSolve()` (forced moves only), `countSolutions()`. Exports `UNK/KEEP/REMOVE`. BigInt bitsets for subset-sum. |
| `js/generator.js` | Puzzle generator: solution mask → Voronoi-BFS regions → values (1–9, oder **„Big Numbers" 10–19** via `opts.bigNumbers`) → targets → uniqueness/no-guess check via solver. Seeded RNG (`mulberry32`). **Big-Numbers-Modus**: `assignValues`/`countSingleDigitSums` sind auf `[minVal,maxVal]` parametrisiert (klassisch [1,9] bleibt bitgenau; der Solver ist ohnehin wertunabhängig via Teilsummen-Bitsets). Foothold-Kriterium bleibt „Summe im Einzelwert-Bereich" — bei 10–19 erzwingt das exakt eine Zelle (2·min>max). Für ALLE Feldgrößen 6×6–14×14 freigeschaltet (`bigNumbersAllowed` in config.js, per Spike verifiziert: überall eindeutig & ratefrei, 14×14 ~1,3 s Ø off-thread); Umschalter im Solo-Setup (`state.sel.bigNumbers`), Brett trägt `.board.big-num` (zweistellige Werte kleiner). `puzzle.minVal/maxVal/bigNumbers` mitgeführt. **Markierungs-Sichtbarkeit**: `remapColorsForMarkVisibility` (rein, unit-getestet) lenkt Cage-Farben um, die einer SPIELERFARBE zu ähnlich sind (`markOnRegionDist` in config.js, Schwelle 160 — kalibriert an COOP_COLORS; Pink-Ring auf Plum-Cage lag bei ~139) — läuft CLIENTSEITIG in app.js (`applyMarkSafeRegionColors`: beim Brettaufbau, per watch bei coopMyColor-/boardPalette-Änderung — der Custom-Farbwähler schreibt via v-model AM setSetting vorbei — und bei neuen Coop-Spielern), mit den EFFEKTIVEN (Paletten-transformierten) Farben; jedes Gerät färbt für seine eigenen Spielerfarben sicher, das Puzzle bleibt identisch. For `maxTier3Steps:0` difficulties it solves WITHOUT hypothesis (much faster reject of bad candidates — key for large boards like 13×13). |
| `js/genworker.js` | ES-module Web Worker that runs `generatePuzzle()` off the main thread via **`generateAsync(opts)`** Promise generation (correlated by `reqId`). Used **on-demand** by Solo `newGame` (loading overlay) and Coop-host/Race/Team (ready-lobby loading bar). Falls back to synchronous generation if the worker is unavailable. **No background prefetch** — puzzles are generated only when a round starts (the old startup-time pre-generation of all difficulties was removed; it bogged down slower devices). |
| `js/training.js` | Tier-1 only solver variant, returns one explained step at a time for tutorial mode. |
| `js/storage.js` | All `localStorage`. Keys prefixed `cns_`. Solo/coop active games in separate slots. **Sicherheitskopien statt stillem Löschen**: `cns_conflict_backup` (beim Merge/Konfliktdialog verdrängter Daten-Stand) + `cns_active_game_backup` (verdrängter Solo-Spielstand), beide GERÄTELOKAL (nie synct); UI zum Einsehen/Einspielen in Einstellungen ▸ Daten (`safetyBackups`/`restoreSafetyBackup` in app.js — Einspielen TAUSCHT, der aktuelle Stand wird selbst zur Kopie). **Aktivspiel-Schutz beim Import**: `importFromFile` (auch der Cloud-`applyCloud`-Pfad) überschreibt die Slots `activeGame`/`activeGameCoop` NIE blind — `pickActiveGame(local, imported)` (rein, unit-getestet) lässt den jüngeren Stand (`ts`) gewinnen und bewahrt ein lokales laufendes Spiel vor `null`/älteren Snapshots (Symptom vorher: „nach einem Update ist mein Spiel weg“, weil der Kaltstart-Reconcile den Snapshot eines Zweitgeräts ohne offene Partie übernahm); ein verdrängter Solo-Stand landet im `ACTIVE_GAME_BACKUP`. **Gelöste Stände nie fortsetzbar**: `snapshotSolved(g)` (rein, unit-getestet) erkennt ein bereits vollständig korrekt gelöstes Brett; `pickActiveGame` behandelt es wie „nicht vorhanden" und `refreshResume` (app.js) räumt einen solchen Slot beim Start auf — sonst tauchte ein abgeschlossenes Spiel als „Fortsetzen" auf und lud ein interaktionsloses 100%-Brett (Win feuert nur auf einen Zug, nicht beim Laden; entstand u.a. durch einen früher an Infinity gescheiterten „done"-Upload, der einen alten „playing"-Cloud-Stand mit gelöstem Brett hinterließ). **Geldverlauf**: `grantCurrency`/`spendCurrency` schreiben zusätzlich ein Transaktionslog (`cns_wallet_log`, Einträge `{id, ts, amount, reason, meta}` — meta = Herkunfts-Details wie Schwierigkeit/Modus/Basis/Multiplikator/gameId). Das Log ist NUTZDATEN-KEY und SYNCT mit (in `collectExportData`; Import/`mergeSnapshots` = Union-Merge nach id via `mergeWalletLogs`, Kappung 120). **Falsch-Geschenk-Fix**: `applyCloudWallet(w, cloudLog)` und `applyCloud` buchen nur noch den durch fremde Log-Einträge NICHT erklärten Rest einer Saldo-Differenz als `gift`/`adminRevoke` (`unexplainedWalletDelta`, rein/unit-getestet) — erspieltes Geld anderer Geräte bringt seine `win`-Einträge selbst mit und erschien vorher beim Login pauschal als „Admin-Geschenk". **Geldverlauf**: `grantCurrency`/`spendCurrency` schreiben zusätzlich ein FIFO-Transaktionslog (`cns_wallet_log`, ±-Betrag/reason/balance, max 60, rein lokal/UI); `applyCloudWallet` bucht die Differenz als `gift`/`adminRevoke`; `noteWalletTransaction(amount, reason)` bucht rein ins Log ohne Saldo-Änderung (z.B. Admin-Selbstgabe, deren Saldo schon per Snapshot-Import gesetzt wurde). UI: `openWalletLog()`/Modal via Guthaben-Chip im Shop; ein Sieg-Eintrag ist AUFKLAPPBAR und schlüsselt den Multiplikator auf (`walletEntryFactors`: Basis · jeder Faktor einzeln · Gesamt), Kontext-Zeile `walletEntryDetail` = Schwierigkeit · Modus. |
| `js/session.js` | **Multi-Device-Konsistenz des aktiven Solo-Spiels** (reine Logik, unit-getestet). Angemeldete Accounts halten die LAUFENDE Partie zusätzlich als EINE autoritative Cloud-Session (`/users/{uid}/session` = `{gameId, rev(monoton), status, deviceId, updatedAt(serverTime), appBuild, schema, payload}`). `decideSessionSync({local, cloud, selfDevice, knownSchema})` liefert die Aktion beim Sichtbarwerden/Live-Event: `inSync`/`uploadLocal`/`takeCloud`/`takeCloudReadonly`/`defunct`(+`backupLocal`)/`reloadRequired`. Kernregeln: selbe `gameId` → `rev` entscheidet (Cloud beendet/weiter → `defunct`/übernehmen; lokal höher → hochladen); andere `gameId` → jüngerer `updatedAt` gewinnt, Verlierer als Backup (nie still gelöscht); Cloud-`schema` > eigenes → `reloadRequired` (kein payload-Parse). `shouldGrantReward(gameId, completed)`/`nextRev`. **Transport in account.js** (`readSession`/`writeSession` via RTDB-**Transaktion** = Compare-and-Set: veraltetes Gerät kann neueren Stand nie überschreiben; Auto-Handoff = neuester Schreiber besitzt/`watchSession` Live-Invalidierung). **Wiring in app.js**: `state.gameId`/`sessionRev`/`sessionReadonly`/`deviceNotice`; `pushSession(status)` an den `syncCloudNow`-Eventpunkten (Start/Pause/Verstecken/Ende); beim Sichtbarwerden `reconcileSession()` STATT blindem Upload (der überschrieb sonst einen neueren Fremdstand); Übernahme-Banner (`.device-notice`, `device.*`-i18n) + `reclaimSession()`; Belohnungs-Idempotenz am Sieg via `completedGames` (cloud-synce Union, keine Doppel-Coins). storage.js: `deviceId()` (per Gerät, NIE synct), `completedGames`/`markGameCompleted`/`mergeCompletedGames`, `ACTIVE_GAME_BACKUP`. **Rules:** `/users/{uid}/session` via Owner-Regel abgedeckt (+ `.validate`) — `database.rules.json` muss in der Firebase Console neu publiziert werden. Nur angemeldet; anonym/lokal = ein Gerät. |
| `js/debuglog.js` | Persistent on-device diagnostic log (`localStorage` `cns_debuglog`, 400-entry FIFO). `log(category, message, extra)` — used everywhere for diagnosis. User exports it via Settings ▸ Diagnoseprotokoll (`exportLogToFile()` prepends BUILD + `userAgent`). `app.js initDiagnostics()` captures unhandled errors/rejections (`error` cat), a one-time device/env snapshot (`env`), and aggregated long-task **jank** every 10s (`perf`). **Log only low-frequency events** (start, errors, lifecycle, generation timing, aggregated perf) — `log()` does synchronous `localStorage` I/O, so never call it per-frame/per-tap or it degrades the very performance you're trying to measure. |
| `js/coop.js` | Firebase RTDB transport, lazy-loaded (solo never loads Firebase). Events under `/rooms/{code}/events`, presence via `onDisconnect()`. **`watchConnection(f, cb)`** überwacht die EIGENE Verbindung über `.info/connected` (feuert rein lokal) → `cb(online, isReconnect)` an `handleCoopConnection()` in app.js setzt `state.coop.online`; so sieht auch der Client selbst einen stillen Idle-Disconnect (Chip „offline"), und bei Reconnect wird die eigene Anwesenheit (`selfInfo`) + `onDisconnect` neu gesetzt. **Prävention:** app.js hält während eines laufenden Spiels einen **Screen Wake Lock** (`requestWakeLock`/`releaseWakeLock`, in `navigate()` + bei `visibilitychange`), damit das Gerät nicht in den Standby geht und die PWA (und damit die RTDB-Verbindung) gar nicht erst abbricht. **Reconnect-Heilung (warm):** nach Reconnect liest `watchConnection` `meta/hostId` neu (Rolle ggf. → guest, kein Doppel-Host) und `handleCoopConnection` (app.js) meldet IDENTITY erneut; der Host nimmt Rückkehrer zusätzlich direkt aus `players/`-onChildAdded auf (`hostRegisterPlayer`) — verhindert die „halbe Lobby" (Partner entfernte einen still Abgerissenen aus dem Roster, `coopSend` blockte dann für immer). `coopSend` puffert Züge während Roster-Lücken (Outbox, Flush bei Heilung). **Kalter Rejoin** (`rejoin({afterEventKey})`): Event-Listener setzt via `orderByKey().startAfter(lastEventKey)` HINTER dem zuletzt verarbeiteten Event auf, statt die Historie zu replayen (das alte INIT überschrieb sonst den Spielstand + reaktivierte die Bereit-Lobby); der Anker wird mit der Coop-Session (storage.js, TTL 12h) persistiert, „Coop fortsetzen" erscheint nur bei gültiger Session. **Raum-Lebensdauer/Heilung:** `leave({keepRoom})` räumt den Raum NIE auf, wenn der Verlassende eine fortsetzbare Session gesichert hat (quitToHome mitten in der Runde → `coopReset({keepRoom:true})` in app.js), und löscht sonst nur nach serverseitig bestätigter Leere (`runTransaction`, bricht ab sobald irgendein Spieler eingetragen ist) — der alte get()+remove-Pfad löschte den Raum unter einem nur STILL abgerissenen Partner weg („Raum existiert nicht mehr" für beide). `rejoin()` akzeptiert auch einen meta-losen „Torso"-Raum (stellt meta wieder her, solange Spieler da sind); wird der EIGENE players-Eintrag extern entfernt (onChildRemoved der eigenen uid), heilt `healSelfPresence` Anwesenheit + meta sofort; `hostGame` räumt beim Neu-Hosten den GANZEN Alt-Raum ab (nicht nur events). **Frischer Beitritt replayt die Historie NICHT mehr komplett** (spielte sonst das STATUS „won" einer früheren Runde als Sieganimation ab + reaktivierte alte Lobbys): `computeJoinAnchor(events)` (rein, unit-getestet) setzt bei OFFENER Runde exakt ab deren INIT auf (Beitretender rekonstruiert den Rundenstand deterministisch und spielt sofort mit), sonst hinter das letzte Event (nächstes INIT/START kommt live). handleCoopMsg (app.js) verwirft Spielzustands-Events (MOVE/STATUS/…) ohne geladenes Brett bzw. STATUS in der Bereit-Lobby. **Multiplayer-Chat** (`MSG.CHAT`): Textchat für ALLE Modi (Coop/Race/FFA/Team) über die room-weiten Events (`Coop.send`, im Team-Modus bewusst room-weit statt teamEvents → erreicht alle 4). app.js: `state.chat` (ephemer, pro Partie via `resetChat()` in `loadPuzzleIntoState` geleert), `sendChat`/`receiveChat`/`pushChatMessage`/`openChat`/`toggleChat`, `isMultiplayer()`; Absender empfängt sein Event NICHT zurück (coop.js filtert `author===uid`) → eigene Nachricht lokal einfügen. UI: Chat-Button in der Spiel-Topbar (`.chat-btn` + Ungelesen-Badge), Bottom-Sheet-Modal `.chat-modal` (eigene Nachrichten rechts/Akzent, fremde links mit Name in Spielerfarbe + Badge). Icons `chat`/`send` in icons.js, `chat.*`-i18n in allen Sprachen. |
| `js/coop-timer` (in `coop.js`/`app.js`) | Geräteübergreifend synchroner Spieltimer: `coop.js` beobachtet `.info/serverTimeOffset` und exportiert `serverNow()` (= `Date.now()+offset`, Solo/ohne Firebase = `Date.now()`). `app.js` `gameNow()` nutzt `serverNow()` in Coop/Race/Team und stempelt damit `state.startTime` (in `startCoopRound`) UND liest die Zeit aus (in `startTimer`/`pauseGame`/`resumeFromPause`) — dieselbe Uhr auf beiden Seiten, sonst ergab die Host-Uhr-Abweichung bei Gästen eine falsche/negative Spielzeit. `elapsed` ist zusätzlich `Math.max(0, …)`-geklemmt. |

| `js/account.js` | **Optional** accounts (email+username+password) + cloud-sync, lazy like coop (anonymous-first: no login ⇒ never loaded, app stays local). Pure validators (`normalizeUsername`/`isValidUsername`/`isValidEmail`/`passwordIssue`/`usernameKey`/`errKey`) are unit-tested; auth/sync wrap Firebase. `/users/{uid}` tree (profile/role + `data` snapshot); username uniqueness via `/usernames/{key}`. Sign-in/up/out/delete **reload the page** so the shared Firebase/auth singleton re-inits with the correct uid (both sign-in AND sign-up persist `profile.accountId` so the logged-in UI survives the reload). **Auto-sync** (`syncNow`/`scheduleSyncUp`, `lastSyncAt` in own key `cns_last_sync`, shown in the account card): **sofortiges** `syncCloudNow(reason)` (app.js → `doSyncNow`, NICHT entprellt) bei Hauptevents — Spielstart (solo `finishNewGame`/coop `startCoopGame`), Pause, Sieg, Niederlage, App-Verstecken/`pagehide`; periodisch alle 30s NUR außerhalb einer Partie (`!gameSessionActive()`). `scheduleSyncUp` (30s-Min-Gap) bleibt für Nicht-Event-Änderungen (z.B. Settings). Grund: der frühere 30s-Debounce verlor den Upload, wenn die App direkt nach einem Sieg geschlossen wurde → fertiges Spiel tauchte wieder als „Fortsetzen" auf; die Events sind selten (kein Per-Zug-Sync), der iOS-Absturz kam von Animationen, nicht der Sync-Rate. **Conflict-safe reconcile at startup** (`reconcile`/pure `decideSync`): compares local vs cloud via a data-revision (`cns_data_rev`, bumped on every user-data `save()`) against a baseline (`cns_synced_rev`) — decisions `uploadLocal`/`takeCloud`/`inSync`/`merged`/`conflict`. **Divergenz wird INHALTLICH aufgelöst** (formale Rev-Divergenz entstand ständig durch Kleinkram wie den Streak-Tages-Rollover auf einem Zweitgerät): das reine, unit-getestete `mergeSnapshots(local, cloud)` führt alles Mergebare verlustfrei zusammen (Union für Inventar/Erfolge/completedGames, Maximum für Siege-/Partien-/Streak-Zähler, Bestwert für Bestzeiten, Verlauf-Union; Kleinigkeiten wie Settings/Flags folgen der jüngeren Seite) → `decision:'merged'`, still + `safeReload`. Der **Versions-Mismatch-Dialog** erscheint NUR noch bei abweichendem Geld-Saldo (`walletBalanceDiffers` — Geld kann durch Käufe legitim sinken, automatisches Raten würde Käufe zurückerstatten); die Wahl in `resolveConflict(choice)` entscheidet dann NUR das Guthaben, der Rest bleibt gemergt. **Local is never silently overwritten**: empty cloud ⇒ upload local; die unterlegene Seite landet immer im `saveConflictBackup`. Sign-in defers all merging to this startup reconcile (no overwrite in `signIn`). Inventory/wallet/profile model lives in `storage.js` (`grant/spend/mergeInventory`, etc.). Inventory syncs via a dedicated **union-only** node `/users/{uid}/inventory` (gifts never clobbered by a client upload); rest of the snapshot under `/users/{uid}/data`. **Username änderbar**: `changeUsername(newName)` hängt den `/usernames`-Index um + aktualisiert `profile.username`. **Live-Geschenke**: `watchGifts` beobachtet eigenes `/inventory` + `data/wallet` (angemeldet) — Admin-Geschenk/-Entzug/Guthaben sofort nutzbar ohne Neustart (`reconcileInventoryFromCloud`/`applyCloudWallet` in storage.js; Selbst-Unlocks bleiben, gift-Einträge folgen der Cloud). **Admin** (`role==='admin'`): `adminFindUser`/`adminGrantItem`/`adminRevokeItem`/`adminSetRole`/`adminGrantCurrency`/`adminSetCurrency`/`adminSetUsername`/`adminSetProfileField` (beliebiges Profilfeld außer role/username)/`adminSendPasswordReset` (Reset-Mail; direktes Setzen geht per Firebase nicht) — gated in UI (In-Game-Badge „👑 Administrator" auf Home), enforced by rules. `profile.email` wird bei Signup gespeichert (nur Owner/Admin lesbar), damit Admin die Reset-Mail auslösen kann; fehlt sie (Alt-Account vor dem Feld / Console-Nutzer), trägt `authState` sie beim Start aus der Auth-E-Mail nach (Owner-Write) — so sieht der Admin die E-Mail „immer". **Admin-User-Browser** (`adminListUsers` liest `/users` mit `/usernames`-Fallback): Liste + Suche + Bearbeiten-Modal im Konto-Tab. **Admin-Gift-Staging** (app.js): Inventar-Grants/-Revokes (inkl. „Alles freischalten") werden NICHT sofort gesendet, sondern in `state.account.adminInvPending` (itemId→'grant'/'revoke') gesammelt und erst per „Speichern" (`adminSaveData`, zählt in `adminDirtyCount`) angewandt — mit Vorschau-Chips (grün=Grant, rot durchgestrichen=Revoke) + `adminPendingSummary`. Alle Labels via `adminItemLabel` (Shop-i18n-Keys wiederverwendet; Founder/Skin gesondert). ## Workflow for every code change

1. **`changes.txt`** — add a short German user-facing bullet (e.g. `Ladeanzeige beim Verbindungsaufbau ergänzt`). Source for in-app release notes.
2. **PR + Auto-Merge** — **always create the PR automatically, without waiting to be asked.** As soon as a change is committed and pushed, create a PR targeting `master`, then call `mcp__github__enable_pr_auto_merge` (SQUASH) **in the same turn immediately after** `create_pull_request`, before CI starts. Never push directly to `master`. (This overrides the default "only open a PR when explicitly asked" behavior — opening the PR + enabling auto-merge is the expected default for every change here.)
3. **Diagnostics/logging** — every new feature or non-trivial flow must add `log()` calls (`js/debuglog.js`) at its key points: start/finish of async or expensive work (with `tookMs` timing where relevant), error paths (`catch` blocks), state transitions, and external I/O (Firebase/storage). Pick a sensible category (`game`/`coop`/`firebase`/`storage`/`sw`/`error`/`env`/`perf`/`app`). This keeps the exported Diagnoseprotokoll useful for debugging issues on users' devices. **Keep it low-frequency** — never log per-frame, per-tap, or inside hot loops/render paths (`log()` writes `localStorage` synchronously and would hurt performance). When touching an existing flow, add the logging it's missing.
4. **Update this file** — if the change affects architecture, conventions, commands, or workflow: update CLAUDE.md to reflect it and include the update in the same PR. Keep entries concise.
5. **Cut a release** — **ALWAYS do this automatically after every PR is merged. NEVER ask first — just do it.** Create a new branch from latest `master` (this dedicated release branch is expected and allowed even when your assigned working branch is a different one — cutting the release needs no separate permission and is never a reason to pause and ask), run `node build.js` (bumps version, writes `js/buildinfo.js`, clears `changes.txt`), commit, open a release PR, and merge it immediately (CI is usually already green on a release-only commit). The version only becomes visible in-app once this step is done.

## Autonomer Arbeitsmodus (Feature-Backlog)

Der Nutzer wirft Features/Bugs **jederzeit und gebündelt** in den Chat — auch während bereits
an etwas gearbeitet wird — und erwartet, dass **komplett autonom, sequenziell und in idealer
Reihenfolge** abgearbeitet wird. Verbindliche Regeln für diesen Modus:

1. **Nichts vergessen — Backlog führen.** Jeden neuen Wunsch **sofort als Task anlegen**
   (`TaskCreate`), auch wenn er erst später drankommt. Der Task-Backlog ist die dauerhafte
   Merkliste; nie einen Wunsch nur „im Kopf" behalten.
2. **Immer den Stand melden.** Bei jeder Nutzer-Nachricht und an Meilensteinen kurz zeigen:
   **✅ durch · 🔄 in Arbeit · ⏭️ als Nächstes** (inkl. Reihenfolge). Der Nutzer soll nie raten
   müssen, was passiert.
3. **Ideale Reihenfolge selbst wählen & umsortieren.** Reihenfolge nach Abhängigkeit/Aufbau
   bestimmen (baut B auf A auf, kommt A zuerst) — Tasks dürfen umgeschoben werden. Kurz begründen.
4. **Parallel zur Pipeline arbeiten.** Während eine CI-Pipeline läuft, **nicht idlen**: schon die
   nächste Aufgabe entwickeln. Dafür einen **git-Worktree** (isolierte Kopie) nutzen, damit der
   offene PR-Branch sauber bleibt; nach dem Merge des laufenden PRs den Zweig zurücksetzen und die
   vorbereitete Arbeit übernehmen (cherry-pick).
5. **Ein PR pro Feature**, jeweils voller Workflow (changes.txt → PR + Auto-Merge → grünes CI →
   Merge → **Release cutten**). Nie mehrere unfertige Features in einem PR mischen.
6. **Nur bei echtem Blocker fragen.** Sonst „nach bestem Wissen & Gewissen" entscheiden, die
   Annahme im PR/Chat protokollieren und weitermachen. Fragen sammeln statt einzeln unterbrechen.
7. **Limit/Reset überbrücken.** Wird ein Nutzungs-/Zeitlimit erreicht, per `ScheduleWakeup`
   (o.ä.) einen Wecker auf den Reset-Zeitpunkt stellen und **automatisch weitermachen** — der
   Nutzer soll nur bei Bedarf reinsprechen müssen.
8. **Abschluss-Zusammenfassung.** Wenn der Backlog leer ist (oder auf Nachfrage): kompakte
   Übersicht aller erledigten Punkte + **offene Fragen/Klärungsbedarf/Wünsche** an den Nutzer.

## Key conventions

- **Offline-First** (`sw.js` + `state.net` + Konflikt-Reconcile): Die PWA ist offline (Flugmodus) voll spielbar. **`sw.js`** liefert die App-Shell **cache-first** mit Navigations-Fallback auf `index.html` (Home-Icon startet offline zuverlässig), alle gleich-origin GET-Assets als **stale-while-revalidate**, Fremd-Origin (Firebase) wird durchgereicht; Precache **einzeln** (`Promise.allSettled`) + **atomarer Swap** (alte Caches erst löschen, wenn der neue die Shell hat) → kein Aussperren nach misslungenem Update. **JEDES neue `js/`-Modul MUSS in die `ASSETS`-Liste** (sonst offline nicht ladbar — passierte mit `session.js`). Cache-Version bumpt `build.js` pro Release. **Manuelle Update-Prüfung**: Klick auf die Version im Hauptmenü → `checkForUpdate()` (app.js) stößt `reg.update()` an; wartet danach ein neuer Worker, bietet ein Dialog den Neustart an (`restartForUpdate` → `SKIP_WAITING`-Message an sw.js → `skipWaiting()` → `safeReload`). Weiterhin KEIN automatischer Laufzeit-Update-Flow — ohne Nutzeraktion übernimmt eine neue Version erst beim nächsten Kaltstart. **`state.net`** (`'online'|'offline'|'reconnecting'`, `initConnectivity()` aus `navigator.onLine`+Events+visibilitychange) → `isOnline()`; netzabhängige Aktionen (Coop/Race/Freunde) offline `disabled` + Guard statt Firebase-Timeout; Home zeigt `.offline-chip`; `onReconnect()` triggert `reconcileSession`+`doSyncNow`. Solo/Training sind komplett lokal. **Versions-Mismatch**: `account.isDivergent()` (rein, unit-getestet) erkennt formale Divergenz (lokal UND Cloud seit `syncedRev` geändert); `reconcile()` löst sie INHALTLICH auf — verlustfreier Auto-Merge (`mergeSnapshots`, s.o.) ⇒ `decision:'merged'` still; NUR bei abweichendem Geld-Saldo (`walletBalanceDiffers`) ⇒ `decision:'conflict'` → app.js `openVersionMismatch`/`resolveVersionMismatch(choice)` zeigt den `.mismatch-modal`, die Wahl betrifft NUR das Guthaben (Rest bleibt gemergt); `account.resolveConflict(choice)` wendet an und sichert die unterlegene Seite als Backup (`saveConflictBackup`, nie still gelöscht), dann `safeReload`. **Grenze:** der Erststart muss einmal online sein (GitHub Pages cached erst dann).
- **Board-Render-Performance (KERNREGELN):** Die App ist EINE große Vue-Komponente — JEDE reaktive Änderung rendert das komplette Spiel-Template inkl. aller Zellen neu. Deshalb: (1) **Hochfrequente reaktive Writes vermeiden** — der Spieltimer schreibt `state.elapsed` nur beim SEKUNDENwechsel (native 250-ms-Abtastung bleibt; vorher: 4 Voll-Renders/s, deren Kosten mit den Markierungen wachsen → auf älteren Geräten Sekunden Tap-Latenz im Spätspiel; `win()`/`lose()` stempeln die exakte ms-Zeit selbst). (2) **`cellStyle` liefert bei unveränderten Eingaben DIESELBE Objekt-Referenz** (`cellStyleCache`, Reset in `loadPuzzleIntoState`) — sonst setzt Vue alle CSS-Custom-Properties (`--rc-*`/`--markcol`) für jede Zelle bei jedem Render neu (`setProperty` war ein Top-Profil-Posten). (3) **NIE `filter` (drop-shadow) auf ein dauerhaft ANIMIERTES Element** — der Skin-Glow liegt als box-shadow auf dem STATISCHEN `::before` unter dem rotierenden `::after` (Filter auf ~165 animierten Layern sättigte den iOS-Compositor: halb gerenderte Kacheln, hängende Taps); `.board.skin-freeze` (bei Pause) hält die Rotation an, damit der `backdrop-filter`-Blur des Pause-Overlays nicht pro Frame neu blurt. (4) **Kein erzwungenes Layout pro Mutation** — `scheduleScrollLockUpdate` (scrollHeight-Messung via MutationObserver) ist auf 250 ms gedrosselt. (5) Aria-Labels/`pulseEdges` sind gecacht bzw. mit Früh-Ausstieg. Im Brett-Template + `cellClasses` NIE pro Zelle den Käfig/Zeile/Spalte neu scannen — `regionResolved`/`rowResolved`/`colResolved` sind O(Zellen) und liefen sonst ×196 pro Render (14×14 „R.I.P." → Ruckeln/Hitze gegen Spielende). Stattdessen die cachenden Computeds `resolvedRegionSet`/`resolvedRowSet`/`resolvedColSet` + die O(1)-Render-Helfer `regionResolvedR`/`rowResolvedR`/`colResolvedR` nutzen (rechnen nur bei `state.marks`-Änderung neu). Die rohen `*Resolved`-Funktionen bleiben nur für die punktuelle Spiellogik (Vorher/Nachher-Prüfung in `setMark`). `.cell` trägt `contain: layout style`.
- All UI strings via `t()` from `js/i18n/index.js` — no hardcoded text.
- **Nie rohe Backend-/DB-Keys in der UI.** Alles, was der Nutzer sieht (Labels, Buttons, Auswahl-/Dropdown-Werte, Titel), muss menschenlesbarer Klartext in korrektem Deutsch sein — nie ein roher Feld-/Schlüsselname aus dem Datenmodell. Das gilt besonders fürs **Admin-Panel**: der Daten-Editor übersetzt jedes Snapshot-Feld über `admin.dict.f.<pfad>.l/.d` (de-only, s.u.), Sektionen über `admin.sec.*`, Enum-Werte über `admin.dict.o.*`; Profilfeld-Auswahl über `adminProfileFieldLabel`, JSON-Untereditor-Titel über `adminPathLabel`. **Wird ein neues Datenfeld / eine neue Einstellung / ein neuer Enum-Wert eingeführt, IMMER im selben PR den passenden `admin.dict`-Eintrag ergänzen** (sonst fällt die UI auf den rohen Key zurück). Der i18n-Paritätstest schließt `admin.dict.*` bewusst aus (Admin-Panel ist de-only) — neue `admin.dict`-Keys brauchen also nur `de.js`, alle anderen sichtbaren Strings weiterhin alle Sprachen.

HINTS`) und läuft in `Coop.send`/`sendTeamEvent` sowie `writeSession`. JEDER neue Firebase-Write, der Spielstand-Zahlen sendet, MUSS hier durch.
