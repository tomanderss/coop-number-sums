# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

A Number Sums logic puzzle PWA (Progressive Web App) — playable offline, dark mode by default, with multiple grid sizes and difficulty levels. Deployed via GitHub Pages. Also distributed as native iOS/Android apps via Capacitor (wrapping the unchanged web app).

**No build tooling for app code.** Vanilla JS + Vue 3 loaded via ESM from `js/vue.esm-browser.prod.js`. No bundler, no transpiler, no TypeScript.

## Commands

```bash
# Run all tests (unit + e2e)
npm test

# Unit tests only (Node built-in test runner)
npm run test:unit

# E2E tests only (Playwright, starts a local http.server on port 8099)
npm run test:e2e

# Run a single unit test file
node --test test/unit/solver.test.js

# Run a single e2e test file
npx playwright test test/e2e/gameplay.spec.js

# Sync web assets to www/ then run cap sync (needed before any native build)
npm run cap:sync

# Regenerate icons
node create-icons.js

# Cut a new release (bumps version, updates js/buildinfo.js and sw.js, empties changes.txt)
node build.js
node build.js --major  # increment major version, reset minor to 0
```

There is no dev server script in package.json — use `python3 -m http.server 8099` directly (Playwright's webServer does this automatically during e2e tests). On Windows: `start-server.bat`.

## Versioning and releases

Version is stored in `.release-counter` (e.g. `0.115`), **not** derived from git commit count. `build.js` increments the minor number, writes `js/buildinfo.js` (auto-generated, do not edit), bumps the cache name in `sw.js`, creates a `version-X.Y.txt` marker file, and empties `changes.txt`. Before a release, add bullet points to `changes.txt` — they appear in the in-app "What's new" popup and changelog.

## Architecture

### Module structure (`js/`)

| File | Role |
|---|---|
| `config.js` | All static config: difficulty definitions, color palettes, `DEFAULT_SETTINGS`. No logic. |
| `solver.js` | Deduction engine. `logicalSolve()` finds only forced moves (no guessing). `countSolutions()` verifies uniqueness. Exports `UNK`/`KEEP`/`REMOVE` constants. Uses BigInt bitsets for subset-sum reachability. |
| `generator.js` | Puzzle generator. Builds a solution mask → places colored regions (Voronoi BFS) → assigns cell values (1–9 always) → computes targets → verifies human-solvability and logical uniqueness via `solver.js`. Seeded RNG (`mulberry32`). |
| `training.js` | Replicates only Tier-1 logic from `solver.js` to produce one explained step at a time for tutorial mode. |
| `storage.js` | All `localStorage` access. Keys are prefixed `cns_`. Solo and coop active games use separate slots. 3-slot rolling backups. |
| `coop.js` | Firebase RTDB transport (lazy-loaded — solo players never load Firebase). Events stored under `/rooms/{code}/events`, presence via `onDisconnect()`. |
| `firebase.js` | Firebase init (anonymous auth + RTDB). Lazy-loaded by `coop.js`. |
| `achievements.js` | Achievement definitions + pure `evaluate()` function. No persistence — storage handled by `storage.js`. |
| `streak.js` | Single helper `todayDateStr()`. |
| `debuglog.js` | In-memory log, exportable as a file for diagnosing connection issues. |
| `buildinfo.js` | Auto-generated by `build.js`. Contains `BUILD` (version string) and `CHANGELOG`. |
| `i18n/index.js` | Runtime i18n without a build step. Add a language: create a new file in `js/i18n/`, import it in `index.js`, add to `MESSAGES` and `SUPPORTED_LOCALES`. Fallback locale: `de`. |
| `app.js` | Vue 3 app. Single large `reactive` state object (`state`) covering all screens (home, setup, game, settings, stats). Also exposes `window.__cns = { state, onCellTap, isSolved }` on localhost for E2E tests. |

### Solver tiers

The solver uses three tiers of deduction (see `solver.js`):
- **Tier 1**: Direct constraint — only one cell in a group is unknown, or remaining sum forces all unknowns.
- **Tier 2**: Subset-sum reachability — BitInt bitsets determine which individual cells are forced by the target sum.
- **Tier 2.5**: Killer-Sudoku-style "innie/outtie" — overlap between a region and a row/column with ≥2 shared cells.
- **Tier 3**: Proof by contradiction (hypothetical assignment). `maxTier3Steps: 0` in all difficulties — no puzzle that requires Tier 3 is accepted; the generator retries with a new seed.

### Game modes

- **Solo**: Standard single-player.
- **Coop**: 2–4 players share a live grid via Firebase RTDB. Host generates the puzzle and sends it; events fan out to all listeners. Lives are shared.
- **Race (1v1)**: Two players race on separate grids of the same puzzle. Progress visible as percentage only.
- **Team vs Team**: Two teams compete; each team shares a grid internally. Uses the same Firebase room as coop.
- **Training**: Step-by-step walkthrough using only Tier-1 forced moves with explanations.

### Capacitor (native builds)

`android/` and `ios/` are committed Capacitor project scaffolds. The web app code is unchanged — `scripts/cap-copy-www.js` mirrors the web files to `www/` (gitignored) before `cap sync`. The Service Worker is skipped inside native Capacitor apps (`Capacitor.isNativePlatform()`).

CI builds (`.github/workflows/android-build.yml`, `ios-build.yml`) are manually triggered. Signed Android release bundles require four repository secrets (`ANDROID_KEYSTORE_BASE64`, `ANDROID_KEYSTORE_PASSWORD`, `ANDROID_KEY_ALIAS`, `ANDROID_KEY_PASSWORD`).

## Testing

- **Unit tests** use Node's built-in `node:test` runner (no extra framework). Import modules directly from `js/`.
- **E2E tests** use Playwright against a `python3 -m http.server 8099` instance emulating a Pixel 7 in `de-DE` locale.
- `test/e2e/helpers.js` provides shared helpers. Tests drive the app via `window.__cns` (exposed only on localhost).
- CI runs on push to `master` and all PRs (`.github/workflows/test.yml`).

## Workflow for every code change

**After completing any feature, bug fix, or improvement:**

1. Add a short German bullet point describing the change to `changes.txt`. This file is the source for the in-app release notes — keep entries concise and user-facing (e.g. `Ladeanzeige beim Verbindungsaufbau ergänzt`).

2. Create a pull request targeting `master` and enable auto-merge so it merges automatically once CI passes. Use the GitHub MCP tools (`mcp__github__create_pull_request`, then `mcp__github__enable_pr_auto_merge`). Do not push directly to `master`.

## Key conventions

- All user-visible strings go through `t()` from `js/i18n/index.js` — no hardcoded UI text.
- `changes.txt` accumulates bullet points between releases; `build.js` reads and clears it.
- `database.rules.json` is the Firebase RTDB security rules source — versioned here, applied via the Firebase Console.
- `www/` is generated and gitignored; never edit files there.
- `js/buildinfo.js` is auto-generated; never edit it manually.
